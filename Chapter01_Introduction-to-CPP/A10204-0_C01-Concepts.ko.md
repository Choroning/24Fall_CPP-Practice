## 1. 컴퓨팅과 프로그래밍

### 1.1 소프트웨어의 중요성

현대 사회는 **소프트웨어** 위에서 돌아간다. 스마트폰, 자동차, 은행 시스템, 의료 기기에 이르기까지 소프트웨어가 일상의 거의 모든 면을 제어한다. 하드웨어만으로는 아무것도 할 수 없으며, 소프트웨어가 하드웨어에 목적과 기능을 부여한다.

```
컴퓨터 시스템
├── 하드웨어
│   ├── CPU (중앙 처리 장치)
│   ├── 메모리 (RAM)
│   ├── 저장장치 (HDD/SSD)
│   └── 입출력 장치 (키보드, 모니터 등)
└── 소프트웨어
    ├── 시스템 소프트웨어 (운영체제, 드라이버, 컴파일러)
    └── 응용 소프트웨어 (브라우저, 게임, 오피스 등)
```

**프로그램**은 특정 작업을 수행하기 위해 작성된 명령어의 집합이다. **프로그래밍**(코딩)은 프로그래밍 언어를 사용하여 이 명령어를 설계하고 작성하는 과정이다.

> **핵심:** 소프트웨어는 인간의 의도와 기계의 실행을 연결하는 다리이다. 프로그램이 없으면 아무리 강력한 하드웨어도 유용한 일을 할 수 없다.

### 1.2 프로그래밍 언어

**프로그래밍 언어**는 컴퓨터가 처리할 수 있는 명령을 작성하기 위한 형식 언어이다. 하드웨어로부터의 추상화 수준에 따라 분류한다.

| 수준 | 언어 | 예시 | 가독성 | 이식성 |
|:-----|:----|:-----|:-----:|:-----:|
| 저급 | 기계어 | `10110000 01100001` | 매우 낮음 | 없음 |
| 저급 | 어셈블리어 | `MOV AL, 61h` | 낮음 | 없음 |
| 고급 | C, C++, Java, Python | `cout << "Hello";` | 높음 | 높음 |

**기계어** (1세대):
- CPU가 직접 실행하는 이진 코드(`0`과 `1`).
- CPU 아키텍처마다 다르므로 완전히 **기계 종속적**이다.

**어셈블리어** (2세대):
- 사람이 읽을 수 있는 **니모닉**(예: `MOV`, `ADD`, `JMP`)을 사용한다.
- 기계어와 1:1 대응한다.
- **어셈블러**가 기계어로 번역한다.

**고급 언어** (3세대 이상):
- 자연어에 가까운 문법(`if`, `while`, `class`)을 사용한다.
- 하나의 고급 문장이 여러 기계어 명령으로 번역될 수 있다.
- **컴파일러** 또는 **인터프리터**가 기계어로 번역한다.
- 일반적으로 **기계 독립적** — 동일한 소스 코드를 다른 플랫폼에서 컴파일할 수 있다.

### 1.3 컴파일러

**컴파일러**는 고급 언어의 소스 코드 전체를 실행 **전에** 기계어로 번역한다. 생성된 바이너리는 CPU에서 직접 실행된다.

```
소스 코드 (C++)
        │
        ▼
   ┌──────────┐
   │  컴파일러  │  ← 전체 프로그램을 한 번에 번역
   └──────────┘
        │
        ▼
기계어 코드 (이진수)
        │
        ▼
   ┌──────────┐
   │   CPU    │  ← 직접 실행
   └──────────┘
```

| 번역 방식 | 시점 | 언어 | 실행 속도 |
|:---------|:----|:----|:---------|
| **컴파일러** | 실행 전 (전체 한꺼번에) | C, C++, Go, Rust | 빠름 |
| **인터프리터** | 실행 중 (한 줄씩) | Python, JavaScript, Ruby | 느림 |

C++은 **컴파일 언어**이다. 소스 코드가 실행 가능한 바이너리로 완전히 번역된 후에 프로그램이 실행된다.

> **팁:** Java는 하이브리드 방식을 사용한다 — 컴파일러가 바이트코드를 생성하고, JVM이 런타임에 이를 인터프리트(또는 JIT 컴파일)한다.

## 2. C++ 언어

### 2.1 역사

C++은 Bell 연구소의 **Bjarne Stroustrup**가 만들었다. 그는 **1979년**에 Simula의 객체지향 기능에서 영감을 받아 C를 확장하는 작업을 시작했다. 이 언어는 처음에 **"C with Classes"**라 불렸으며, **1983년**에 **C++**로 이름이 변경되었다 (`++`는 C의 증가 연산자 — "C에서 한 단계 진화").

핵심 아이디어: **C++ = C + 객체 지향 프로그래밍(OOP)**.

```
C++ 언어의 역사
──────────────────────────────────────────────────────────────
1967    BCPL (Martin Richards) — 컴파일러 작성을 위한 간략화된 CPL
1969    B (Ken Thompson) — 초기 UNIX용으로 간략화된 BCPL
1972    C (Dennis Ritchie) — 타입이 추가된 B; UNIX 재작성
1979    C with Classes (Bjarne Stroustrup) — C + OOP 기능
1983    C++ — 이름 변경; 최초 공개 배포
1998    C++98 — 최초의 ISO 표준 (ISO/IEC 14882:1998)
2003    C++03 — C++98 버그 수정
2011    C++11 — 대규모 현대화 (auto, 람다, 이동 의미론,
                스마트 포인터, 범위 기반 for, nullptr)
2014    C++14 — C++11의 소규모 개선
2017    C++17 — 구조화된 바인딩, std::optional, 파일시스템
2020    C++20 — 개념(concepts), 코루틴, 범위(ranges), 모듈,
                3방향 비교 (<=>)
──────────────────────────────────────────────────────────────
```

> **핵심:** C++11은 "고전적인 C++"와 "현대 C++"를 나누는 분기점으로 널리 인식된다. `auto`, 스마트 포인터, 람다 식 등의 C++11 기능으로 인해 많은 코딩 관행이 근본적으로 변화했다.

### 2.2 특징

C++은 C의 저수준 성능과 고수준 추상화를 결합한다. 주요 특징은 다음과 같다:

| 특징 | 설명 |
|:----|:----|
| **C 호환성** | 소스 수준 및 링크 수준에서 C와 호환; 대부분의 C 코드가 C++로 컴파일됨 |
| **객체 지향 프로그래밍** | 캡슐화, 다형성, 상속으로 현실 세계의 개체를 모델링 |
| **엄격한 타입 검사** | C보다 강력한 타입 강제; 컴파일 시점에 더 많은 오류를 포착 |
| **인라인 함수** | `inline` 키워드로 호출 지점에서 함수를 확장하여 함수 호출 오버헤드 제거 |
| **제네릭 / 템플릿** | 타입에 독립적인 코드를 작성; 컴파일러가 각 타입에 맞는 특수화 버전을 생성 |
| **연산자 오버로딩** | 사용자 정의 타입에 대해 연산자(`+`, `-`, `<<` 등)를 재정의 |
| **참조(Reference)** | 기존 변수의 별명(`int& ref = x;`); 많은 경우 포인터보다 안전한 대안 |
| **`new` / `delete`** | C의 `malloc`/`free`를 대체하는 동적 메모리 관리 연산자 |
| **STL (표준 템플릿 라이브러리)** | 풍부한 컨테이너(`vector`, `map`), 알고리즘(`sort`, `find`), 반복자 모음 |

**객체 지향 프로그래밍 — 세 가지 기둥:**

```
OOP
├── 캡슐화(Encapsulation)  — 데이터 + 메서드를 묶고, 내부 세부사항을 숨김 (class, private/public)
├── 상속(Inheritance)      — 기존 클래스로부터 새로운 클래스를 파생; 코드 재사용 (class Dog : public Animal)
└── 다형성(Polymorphism)   — 같은 인터페이스, 다른 동작 (가상 함수, 오버라이딩)
```

```cpp
#include <iostream>
using namespace std;

class Animal {
public:
    virtual void speak() { cout << "..." << endl; }
};

class Dog : public Animal {
public:
    void speak() override { cout << "Woof!" << endl; }
};

int main() {
    Animal* pet = new Dog();
    pet->speak();   // 출력: Woof!  (다형성)
    delete pet;
    return 0;
}
```

> **팁:** C++은 절차적, 객체 지향, 제네릭 프로그래밍 등 여러 프로그래밍 패러다임을 지원한다. 모든 프로그램에 OOP를 사용할 필요는 없다.

### 2.3 표준 C++의 중요성

**ANSI/ISO C++ 표준**은 서로 다른 컴파일러와 플랫폼에서 C++ 코드가 일관되게 동작하도록 보장한다. **표준 준수 코드**를 작성하는 것이 이식성의 핵심이다.

| 분류 | 설명 | 예시 |
|:----|:----|:----|
| **표준 프로그램** | ISO 표준에서 정의된 기능만 사용 | `std::cout`, `std::vector` |
| **비표준 프로그램** | 컴파일러별 확장 기능을 사용 | `__cdecl`, `__declspec`, `#pragma once` |

표준 C++ 프로그램은 모든 호환 컴파일러(GCC, Clang, MSVC 등)에서 올바르게 컴파일 및 실행된다. 비표준 프로그램은 특정 컴파일러나 플랫폼에서만 컴파일될 수 있다.

```cpp
// 표준 C++ — 이식 가능
#include <iostream>
int main() {
    std::cout << "Hello" << std::endl;
    return 0;
}
```

```cpp
// 비표준 — 컴파일러별 확장 기능 (MSVC)
void __cdecl myFunc();   // __cdecl은 MSVC 전용 호출 규약
```

> **경고:** 꼭 필요한 경우가 아니라면 컴파일러별 확장 기능의 사용을 피한다. 사용해야 한다면 플랫폼별 `#ifdef` 가드 뒤에 격리하여 나머지 코드의 이식성을 유지한다.

## 3. C++ 표준 라이브러리

C++은 풍부한 표준 라이브러리를 제공한다. **C에서 물려받은 라이브러리**와 **C++ 고유 라이브러리**로 나뉜다.

| 분류 | 헤더 형식 | 예시 | 설명 |
|:----|:---------|:----|:----|
| C 라이브러리 (C++ 형태) | `<cXXX>` (`.h` 없음) | `<cstring>`, `<cstdio>`, `<cmath>`, `<cstdlib>` | `std` 네임스페이스에 래핑된 C 표준 함수 |
| C++ 라이브러리 | `<XXX>` (`.h` 없음) | `<iostream>`, `<string>`, `<vector>`, `<algorithm>` | C++ 고유 기능: 스트림, 문자열, 컨테이너 등 |

**C 헤더와의 주요 차이점:**

1. **`.h` 확장자 없음** — C++ 표준 헤더는 `.h`를 사용하지 않는다. `<iostream.h>`는 비표준이며 더 이상 사용되지 않는다.
2. **`std` 네임스페이스** — 모든 표준 라이브러리 이름은 `std` 네임스페이스에 존재한다. `std::` 접두사 또는 `using` 선언으로 접근한다.
3. **C 헤더의 `<cXXX>` 형태** — C의 `<string.h>`는 C++에서 `<cstring>`이 되며, 함수들이 `std::`에 배치된다.

```cpp
#include <iostream>    // C++ 입출력 스트림 라이브러리
#include <string>      // C++ 문자열 클래스
#include <vector>      // C++ 동적 배열
#include <cmath>       // C 수학 함수 (sqrt, pow 등), std:: 내

int main() {
    std::string name = "C++";
    std::vector<int> nums = {1, 2, 3};
    std::cout << name << std::endl;
    std::cout << std::sqrt(2.0) << std::endl;
    return 0;
}
```

> **핵심:** C++ 프로그램에서는 항상 C 형태인 `<string.h>` 대신 C++ 형태인 `<cstring>`을 사용한다. `<cXXX>` 형태는 이름이 `std` 네임스페이스에 배치됨을 보장한다.

## 4. C++ 개발 과정

### 4.1 빌드 파이프라인

C++ 프로그램을 만드는 과정은 네 가지 주요 단계를 따른다:

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   1. 편집     │───▶│  2. 컴파일    │───▶│   3. 링크     │───▶│   4. 실행     │
│  (소스 코드)   │    │ (.cpp → .obj) │    │ (.obj → .exe) │    │  (실행 & 검증) │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

```
  main.cpp          iostream (헤더)
     │                │
     ▼                │
┌────────────┐        │
│  전처리기    │◀───────┘   (#include 확장)
└────────────┘
     │
     ▼
┌────────────┐
│   컴파일러   │             (C++ 코드 → 오브젝트 코드)
└────────────┘
     │
     ▼
  main.obj           (오브젝트 파일 — 기계어, 아직 실행 불가)
     │
     ▼
┌────────────┐
│    링커     │◀─── 라이브러리 파일 (표준 라이브러리 등)
└────────────┘
     │
     ▼
  main.exe           (최종 실행 파일)
```

| 단계 | 입력 | 출력 | 하는 일 |
|:----|:----|:----|:-------|
| **편집** | — | `.cpp` 소스 파일 | 텍스트 편집기 또는 IDE에서 프로그램 작성 |
| **컴파일** | `.cpp` 소스 | `.obj` / `.o` 오브젝트 파일 | C++ 소스를 기계어로 번역; 문법 및 타입 검사 |
| **링크** | `.obj` 파일 + 라이브러리 | `.exe` 실행 파일 | 오브젝트 파일 결합; 외부 참조 해결 (예: `cout`) |
| **실행** | `.exe` 실행 파일 | 프로그램 출력 | 프로그램 실행 및 결과 확인 |

> **핵심:** 컴파일과 링킹은 별도의 단계이다. 컴파일러는 소스 파일을 한 번에 하나씩 오브젝트 파일로 번역한다. 링커가 모든 오브젝트 파일(과 라이브러리)을 하나의 실행 파일로 결합한다.

### 4.2 오류의 종류

C++ 프로그램의 오류는 **감지 시점**에 따라 세 가지로 분류된다:

| 오류 유형 | 감지 시점 | 원인 | 예시 |
|:---------|:---------|:----|:----|
| **컴파일 오류** | 컴파일 시 | 문법 오류, 타입 불일치, 미선언 식별자 | `;` 누락, 잘못된 타입 변환 |
| **링크 오류** | 링킹 시 | 함수/변수 정의 누락, 해결되지 않은 심볼 | 구현이 없는 함수 호출 |
| **런타임 오류** | 실행 시 | 논리 오류, 불법 연산 | 0으로 나누기, 유효하지 않은 메모리 접근 |

**컴파일 오류:**
```cpp
#include <iostream>
using namespace std;

int main() {
    cout << "Hello" << endl  // 세미콜론 누락 → 컴파일 오류
    return 0;
}
```

**링크 오류:**
```cpp
void myFunction();   // 선언만 하고 정의하지 않음

int main() {
    myFunction();    // 링커가 정의를 찾을 수 없음 → 링크 오류
    return 0;
}
```

**런타임 오류:**
```cpp
#include <iostream>
using namespace std;

int main() {
    int a = 10, b = 0;
    cout << a / b << endl;  // 0으로 나누기 → 런타임 오류
    return 0;
}
```

> **경고:** 컴파일 오류는 컴파일러가 잡아주므로 가장 고치기 쉽다. 런타임 오류가 가장 어렵다 — 프로그램이 성공적으로 컴파일되고 링크되지만, 실행 중에 잘못된 결과를 출력하거나 충돌한다.

## 요약

| 개념 | 핵심 정리 |
|:----|:---------|
| 소프트웨어의 중요성 | 소프트웨어가 하드웨어에 목적을 부여; 현대 생활은 프로그램에 의존 |
| 프로그래밍 언어 | 기계어 (이진수) → 어셈블리어 (니모닉) → 고급 언어 (C, C++, Python) |
| 컴파일러 | 전체 소스 코드를 실행 전에 기계어로 번역 |
| C++ 역사 | Bjarne Stroustrup, 1979/1983; C with Classes → C++; C++98 → C++11 → C++20 |
| C++ = C + OOP | C와 호환; 캡슐화, 상속, 다형성 추가 |
| 주요 특징 | 템플릿, 연산자 오버로딩, 참조, `new`/`delete`, 인라인 함수, STL |
| 표준 C++ | ANSI/ISO 표준이 이식성을 보장; 컴파일러별 확장 사용 자제 |
| C++ 라이브러리 | `<cstring>` (C 계승) vs `<iostream>` (C++ 고유); `.h` 없음; `std` 네임스페이스 |
| 빌드 파이프라인 | 편집 (.cpp) → 컴파일 (.obj) → 링크 (.exe) → 실행 |
| 컴파일 오류 | 컴파일 시 감지되는 문법/타입 오류 |
| 링크 오류 | 링킹 시 감지되는 정의 누락 오류 |
| 런타임 오류 | 실행 시 감지되는 논리 오류 또는 불법 연산 |
